# Command Library

## üßπ Code Quality & Cleanup

### `/cleanup-unused-code` ‚≠ê
**Purpose:** Remove unused imports, commented code, unused variables, console.log  
**Usage:** `/cleanup-unused-code [path] [--remove-console-logs]`  
**Speed:** Fast (30s - 2min)  
**When to use:** Daily cleanup, before commits, after refactoring

**Fixes:**
- Unused imports
- Commented-out code blocks
- Unused variables/functions
- Console.log statements (optional)

---

### `/fix-import-paths` ‚≠ê
**Purpose:** Fix broken imports, standardize import order, convert relative to absolute  
**Usage:** `/fix-import-paths [path]`  
**Speed:** Fast (30s - 1min)  
**When to use:** After moving files, when imports break, code standardization

**Fixes:**
- Broken import paths
- Relative ‚Üí absolute imports (`@/` paths)
- Import order (React ‚Üí External ‚Üí Internal ‚Üí Types)
- Unused imports
- Type-only imports

---

### `/fix-spacing-layout` ‚≠ê
**Purpose:** Convert hardcoded spacing to design tokens, fix layout inconsistencies  
**Usage:** `/fix-spacing-layout [path]`  
**Speed:** Fast (30s - 2min)  
**When to use:** Design token compliance, layout standardization

**Fixes:**
- Hardcoded px values ‚Üí Design tokens
- Inconsistent spacing ‚Üí Standard spacing scale
- Inline style spacing ‚Üí Tailwind classes
- Layout alignment issues

---

### `/refactor-cleanup` ‚≠ê
**Purpose:** Comprehensive refactoring with better cleanup patterns  
**Usage:** `/refactor-cleanup [path]`  
**Speed:** Moderate (2-5min)  
**When to use:** Major refactoring, code quality improvements

**New capabilities:**
- Better unused code detection
- Integration with import fixes
- More aggressive cleanup
- Uses `/cleanup-unused-code` patterns

---

## `/suggest-command` - Context-Aware Suggestions

Get command suggestions based on current file/context.

**Usage:**
```
/suggest-command src/features/email-marketing/pages/FormsPage.tsx
/suggest-command  # Analyzes currently open file
```

**What It Detects:**
- Code quality issues (unused imports, commented code)
- UI component issues (filter bars, tables, forms)
- Design system issues (hardcoded values)
- Testing needs

**What It Suggests:**
- Commands to fix issues (high priority)
- Commands to standardize (medium priority)
- Commands for enhancements (low priority)

# Command Comparison

## Cleanup Commands

### `/cleanup-unused-code` vs `/refactor-cleanup`

| Aspect | `/cleanup-unused-code` | `/refactor-cleanup` |
|--------|------------------------|---------------------|
| **Purpose** | Remove unused code | Comprehensive refactoring |
| **Speed** | Fast (30s-2min) | Moderate (2-5min) |
| **When to Use** | Daily cleanup, before commits | Major refactoring |
| **Best For** | Quick cleanup | Comprehensive improvements |

**Recommendation:**
- **Daily/regular cleanup** ‚Üí `/cleanup-unused-code`
- **Major refactoring** ‚Üí `/refactor-cleanup`

---

### `/cleanup-unused-code` vs `/fix-import-paths`

| Aspect | `/cleanup-unused-code` | `/fix-import-paths` |
|--------|------------------------|---------------------|
| **Purpose** | Remove unused code | Fix and standardize imports |
| **Focus** | Unused code removal | Import path fixes |
| **Best For** | General cleanup | Import-specific issues |

**Recommendation:**
- **General cleanup** ‚Üí `/cleanup-unused-code`
- **Import issues** ‚Üí `/fix-import-paths`
- **Both needed** ‚Üí Run both (or use `/daily-cleanup` workflow)

---

# Quick Start Guides

## Scenario 1: First Day on Project
**Goal:** Get familiar with the codebase and commands

**Steps:**
1. `/test-page-quick` - Verify everything works
2. `/pre-flight-check` - Check environment setup
3. `/find-command test` - Discover testing commands
4. Browse command library - Learn available commands

**Time:** ~5 minutes

---

## Scenario 2: Before Creating PR
**Goal:** Ensure code quality before submitting PR

**Steps:**
1. `/pre-commit-checklist [your-changes-path]`
2. Review changes
3. Commit if all good

**Time:** ~5-10 minutes

**Aliases:**
```bash
/clean [path]
/imports [path]
/spacing [path]
/test
/pr
```

---

## Scenario 3: Fixing a Broken Page
**Goal:** Debug and fix a page that won't load

**Steps:**
1. `/debug-failing-page` - Deep analysis
2. `/fix-page-bugs [page-path]` - Auto-fix common issues
3. `/fix-import-paths [page-path]` - Fix imports
4. `/test-page-quick` - Verify fix

**Time:** ~5-10 minutes

---

## Scenario 4: Standardizing a New Page
**Goal:** Ensure new page follows all standards

**Steps:**
1. Create page structure
2. `/standardize-page [page-path]` - Complete standardization
3. Review changes
4. `/test-page-quick` - Verify

**Time:** ~5-10 minutes

---

## Scenario 5: Daily Development Routine
**Goal:** Start day with clean, working codebase

**Steps:**
1. `/test-page-quick` - Verify nothing broke
2. `/daily-cleanup [your-work-path]` - Clean up yesterday's work
3. Start new work

**Time:** ~2-3 minutes

**Aliases:**
```bash
/test
/daily-cleanup [path]
```

# Best Practices

## ‚úÖ DO

### Daily Development
- ‚úÖ Run `/test-page-quick` every morning
- ‚úÖ Use `/daily-cleanup` at start/end of day
- ‚úÖ Use `/cleanup-unused-code` before commits
- ‚úÖ Use `/fix-import-paths` after moving files

### Before Commits
- ‚úÖ Run `/pre-commit-checklist` before every commit
- ‚úÖ Use `/test-page-quick` to verify changes
- ‚úÖ Use `/design-token-check` for design compliance
- ‚úÖ Review changes before committing

### Code Quality
- ‚úÖ Use `/cleanup-unused-code` regularly
- ‚úÖ Use `/fix-spacing-layout` for design compliance
- ‚úÖ Use `/refactor-cleanup` for major refactoring
- ‚úÖ Chain related commands for efficiency

### UI Standardization
- ‚úÖ Use `/standardize-page` for new/legacy pages
- ‚úÖ Use `/fix-filter-bar` with `/implement-view-modes`
- ‚úÖ Use `/fix-data-table` with `/fix-table-columns` if needed
- ‚úÖ Verify with `/test-page-quick` after fixes

### Command Discovery
- ‚úÖ Use `/find-command` to search for commands
- ‚úÖ Use `/suggest-command` when unsure what to use
- ‚úÖ Browse command library for complete reference
- ‚úÖ Check quick start guides for scenarios

---

## ‚ùå DON'T

### General
- ‚ùå Skip testing after fixes
- ‚ùå Use deprecated commands (`/dead-code-removal`, `/remove-dead-code`)
- ‚ùå Run commands on production without testing
- ‚ùå Ignore error messages

### Workflows
- ‚ùå Skip verification steps in workflows
- ‚ùå Run workflows without reviewing changes
- ‚ùå Use workflows as substitute for understanding

### Code Quality
- ‚ùå Commit without running `/pre-commit-checklist`
- ‚ùå Ignore design token violations
- ‚ùå Leave unused code in commits
- ‚ùå Skip import standardization
