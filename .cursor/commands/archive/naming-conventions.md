## Naming Conventions (Project-Wide)

This document defines canonical naming rules for the entire codebase. Follow these conventions for all new code. For architecture, design tokens, and component standards, always cross‑check the primary docs listed in `DOCUMENTATION_INDEX.md`.

### Folder and Directory Names
- **lowercase-kebab-case** for all folders. Folders must start with a lowercase letter.
- **Feature folders** live in `src/features/[feature-name]/` (lowercase-kebab-case).
- Standard feature subfolders (use only what you need):
  - `components/`
  - `hooks/`
  - `services/`
  - `types/`
  - `utils/`
  - `constants/`
  - `docs/`

Example:
```
src/features/subscription-portal/
├─ components/
├─ hooks/
├─ services/
├─ types/
├─ utils/
└─ docs/
```

### File Names and Symbols
- **React components (files and symbols)**: PascalCase
  - File names must match the exported component (e.g., `SubscriptionCard.tsx` exports `SubscriptionCard`).
  - Use `.tsx` for React components, `.ts` for non-JSX modules.
- **Hooks**: camelCase with `use` prefix (file and symbol)
  - Example: `useSubscriptionData.ts` exports `useSubscriptionData`.
- **Types/Interfaces**: PascalCase; suffix is allowed where clarifying
  - Examples: `SubscriptionCardProps`, `Customer`, `Invoice`.
- **Utilities**: camelCase (file and primary export)
  - Example: `formatCurrency.ts` exports `formatCurrency`.
- **Services**: file camelCase; classes PascalCase; instances camelCase
  - Example: `billingService.ts` exports `BillingService` and/or `billingService`.
- **Constants (variables)**: SCREAMING_SNAKE_CASE
  - Example: `DEFAULT_PAGE_SIZE`, `MAX_UPLOAD_BYTES`.
  - If a dedicated file is used, prefer `constants.ts` or a descriptive `billing.constants.ts`.
- **Enums**: PascalCase enum name, SCREAMING_SNAKE_CASE members.
- **Environment variables**: UPPER_SNAKE_CASE (e.g., `VITE_API_URL`).

### Tests
- Use `*.test.ts` or `*.test.tsx`.
- Prefer colocating tests next to the file under test. `__tests__/` is allowed for complex modules.
  - Examples: `SubscriptionCard.test.tsx`, `utils/formatCurrency.test.ts`.

### Exports
- Prefer **named exports** for shared modules and components.
- Default exports are allowed for leaf-level feature components but avoid in shared libraries.
- Barrel files (`index.ts`) can aggregate named exports within a folder; avoid deep barrels across features.

### Shared vs Feature Code
- Reusable UI → `src/shared/components/ui/` (check first before creating anything new).
+- Reusable hooks → `src/shared/hooks/`
+- Utilities → `src/shared/utils/`
+- Constants → `src/shared/constants/`
+- Do not create cross-feature dependencies. Extract to shared if used by multiple features.

### GraphQL and API
- Operation files: prefer descriptive camelCase (e.g., `fetchSubscriptions.ts`) or `.graphql` where applicable.
- Operation/function names: camelCase (e.g., `getCustomerInvoices`).
- Types generated by codegen should follow generator defaults; wrap with local types only when necessary.

### Styling and Design System (Enforced)
- No CSS files in feature folders.
- No inline styles for tokens; use Tailwind classes that map to design tokens.
- All form labels must use `var(--label-color)` (mapped to `var(--color-ocean-600)` via design system).
- Always consult:
  - `DESIGN_TOKENS_REFERENCE.md`
  - `AVAILABLE_COMPONENTS_CATALOG.md`
  - `src/shared/design-system/guides/*`

### Database vs TypeScript Field Names (Reminder)
- Database fields use Mirabel schema names (PascalCase or specific IDs like `gsCustomersID`).
- In-memory/TypeScript domain models use camelCase and `id`.
- See:
  - `prisma/schema.prisma`
  - `scripts/field-mappings.json`
  - `prisma/middleware/field-mapper.ts`

### Examples
Component file:
```ts
// src/features/subscription-portal/components/SubscriptionCard.tsx
export interface SubscriptionCardProps {
  planName: string;
}

export function SubscriptionCard({ planName }: SubscriptionCardProps) {
  return <div>{planName}</div>;
}
```

Hook file:
```ts
// src/features/subscription-portal/hooks/useSubscriptionData.ts
export function useSubscriptionData() {
  // ...
  return { data: [], isLoading: false };
}
```

Utility with constant:
```ts
// src/shared/utils/formatCurrency.ts
export const DEFAULT_LOCALE = 'en-US';

export function formatCurrency(amount: number, currency = 'USD') {
  return new Intl.NumberFormat(DEFAULT_LOCALE, { style: 'currency', currency }).format(amount);
}
```

### Quick Rules Checklist
- Folders: lowercase-kebab-case, start with lowercase.
- Components: PascalCase (`FileName.tsx` exports `FileName`).
- Hooks: `useThing.ts` exporting `useThing`.
- Types: PascalCase (`Thing`, `ThingProps`, `ThingResponse`).
- Utils: camelCase files and functions.
- Constants: SCREAMING_SNAKE_CASE.
- Tests: `*.test.ts(x)`, colocate when practical.
- Shared code lives under `src/shared/*`; no cross-feature imports.

Keep this document updated when conventions evolve. If in doubt, prefer the existing patterns in `src/shared/design-system/` and the examples already in the codebase.
